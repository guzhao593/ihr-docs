# 配置平台 1.0开发使用文档

# 整体结构

  • cpf-config: 业务配置;
  • cpf-repository: 存储引擎；
  • cpf-rule: 校验引擎;
  • cpf-common: 公共组件;
  • cpf-metadata: 元数据组件;
  • cpf-event: 事件组件；
  • cpf-auth: 数据权限组件;
  • cpf-file-parser: 文件解析工具;
  • cpf-codegeneric: 编码规则工具;
  • cpf-plugin-jiebao: 捷报工具;

整体上基于Springboot 条件化配置的方式进行开发，实现功能按需加载或快速覆写，提升启动性能；

# cpf-config

# 功能描述

  • 提供业务组件、单据设置、业务功能、接口管理、数据迁移、事件监听、多数据源等的配置功能。

# 模块依赖

# 功能详解

# 业务组件

业务组件是从业务中分离抽取出来的公共部分字段形成的一个组件。业务组件可以有多个，它们像是一块块不可或缺的积木，任意组装成一个完整的模型；

# 单据设置

单据设置是由业务组件拼装成的一个基础模板。在单据设置上可以添加一些该模板特有的一些字段，以便被使用；

# 业务功能

业务功能是先选择一个基础模板，在此模板上添加个性化信息的一个业务模型。业务功能主要包含视图配置、存储功能。 从功能的角度看，业务功能又细分为单据模板、台账模板、表单模板。

# 接口管理

接口管理提供统一的外部接口管理。

# 数据迁移

数据迁移是为了方便数据在同一套代码的不同环境之间，数据的复制工作。

# 事件监听

事件监听是将系统中所有的监听器进行了统一的管理。

# 多数据源

多数据源是为了在不同数据库之间来回动态切换的一个功能。

# Aviator 自定义扩展函数功能

固定入参的扩展函数可通过继承 AbstractFunction 重写需要覆盖的call方法，并实现Describable接口返回固定结构的描述信息。将实现注册到Spring容器后，可在表达式引用该函数；动态参数的扩展函数需要继承 AbstractVariadicFunction，并实现variadicCall方法。

@Component
public class StringConcatFunction extends AbstractVariadicFunction implements Describable {

    public static final String NAME = "stringConcatFunction";
    public static final CustomizeFunctionDTO DESCRIPTION = new CustomizeFunctionDTO();

    static {
        DESCRIPTION.setFunctionCode(NAME);
        DESCRIPTION.setFunctionName("字符串拼接函数");
        DESCRIPTION.setParamType(CustomizeFunctionDTO.ParamType.DYNAMIC.getKey());
        DESCRIPTION.setParamItems(ListUtils.EMPTY_LIST);
        DESCRIPTION.setResultType(CustomizeFunctionDTO.ResultType.STRING.getKey());
    }

    @Override
    public String getName() {
        return NAME;
    }

    @Override
    public AviatorType getAviatorType() {
        return super.getAviatorType();
    }

    @Override
    public CustomizeFunctionDTO getDescription() {
        return DESCRIPTION;
    }

    @Override
    public AviatorObject variadicCall(Map<String, Object> env, AviatorObject... args) {

        if(args.length == 0) {
            return new AviatorString("");
        }
        StringBuilder sb = new StringBuilder();
        for (AviatorObject arg : args) {
            Object value = arg.getValue(env);
            if (value != null) {
                sb.append(value.toString());
            }
        }
        return new AviatorString(sb.toString());
    }
}

# 自定义数据装载器

common模块提供了装载器相关接口定义，如自定义程序装载器定义、维度装载定义，并提供了SQL装载器的默认实现。业务需要扩展数据装载器，可通过继承ICustomizeProgramLoader抽象类，编写load方法，实现自定义程序装载器装载逻辑。

TIP

数据装载触发时机是在存储引擎执行存储之前，通过多线程的方式，根据数据模型数据的层级和装载方式进行分组装载。

# 多数据源

配置平台的多数据源实现是基于spring-jdbc 的 AbstractRoutingDataSource，重写了其determineCurrentLookupKey和determineTargetDataSource方法，实现从DataSourceContextHolder 的 InheritableThreadLocal 线程变量中读取当前线程使用的数据源别名，InheritableThreadLocal类型的线程变量使得子线程可以继承父线程的值，保证子线程中数据源不发生改变。

  • 数据源切换实现
private Map<String, DataSource> resolvedDataSources = new ConcurrentHashMap<>();

@Resource(name = "defaultDb")
private DataSource resolvedDefaultDataSource;

private volatile IDataSourceResolver dataSourceResolver;

@Override
protected Object determineCurrentLookupKey() {
    return DataSourceContextHolder.getDataSourceAlias();
}

@Override
protected DataSource determineTargetDataSource() {
    // 获取当前线程datasoureId
    Object lookupKey = determineCurrentLookupKey();
    // 如果为空，返回默认数据源
    if(Objects.isNull(lookupKey)){
        return resolvedDefaultDataSource;
    }
    DataSource dataSource = this.resolvedDataSources.computeIfAbsent((String) lookupKey, o -> {
        if(Objects.isNull(dataSourceResolver) && Objects.isNull(dataSourceResolver = ApplicationContextProvider.getBean(IDataSourceResolver.class))){
            return null;
        }
        return dataSourceResolver.resolve(resolvedDefaultDataSource,lookupKey);
    });
    if (dataSource == null) {
        throw new IllegalStateException("Cannot determine target DataSource for lookup key [" + lookupKey + "]");
    }
    return dataSource;
}

     * 取得当前数据源

    public static String getDataSourceAlias(){

     * 清除上下文数据

    public static void remove(){
        contextHolder.remove();
    }

  • 编程实现数据源切换

    Spring默认的事务机制决定了在同一事务下无法直接切换数据源，为保证切换生效，暂时采用了另起线程执行的方式，进行不同数据源下的业务调用。配置平台提供了DatasoureThreadPool类，用于切换到不同数据源执行。该类重载的execute方法，分别为Supplier和Runable作为入参，对应有返回值和无返回值的使用场景。使用示例如下：

     DatasoureThreadPool.otherDatasource(null).execute(()-> init(reimId, dataModelId, reqParam));

@RegistrableEventListener(description = "【报账单据回调接口】：报账单付款状态变化通知上游业务系统", condition = "!'EXPENSE'.equals(#this.event.source.originSystem)")
public void sendPayStatusToOutSystem(PayStatusEvent event) {

}

# 维度定义

配置平台数据权限控制默认实现从redis获取权限数据。业务可自定义实现权限维度定义相应方法，如权限数据的key前缀、缓存权限数据、获取默认权限数据等，最后注册到spring容器

public interface IDimensionDefinition {

    /**
     * 根据用户获取redis缓存数据key
     *
     * @param account
     * @return
     */
    String getKey(String account);

    /**
     * 维度权限数据key前缀
     *
     * @return
     */
    String keyPrefix();

    /**
     * 获取维度列表
     *
     * @return
     */
    List<Dimension> dimensionList();

    /**
     * 将当前登录用户对应维度权限缓存至redis
     *
     * @return
     */
    boolean saveAuthCache();

    /**
     * 根据维度编码获取当前人在该维度下的默认权限数据
     * @param dimensionCode 维度编码
     * @return 权限数据
     */
    Object getDefaultAuthData(String dimensionCode);

    @Data
    class Dimension{
        String dimensionCode;
        String dimensionName;
    }
}

# 生成编码

按照一定的规则生成唯一编码，规则存储在bd_code_rule表中。

使用步骤：

  1. 插入规则到bd_code_rule表。
  2. 引入cpf-codegeneric依赖包，调用CpfCodeGenerator的updateAndGetCode()方法，把规则编号当作入参，返回一个唯一编码。

代码示例：

    private String getNextCode(String custCode) {
        try {
            String code = cpfCodeGenerator.updateAndGetCode(custCode);
            if (StringUtils.isBlank(code)) {
                throw new ValidateException("编码生成失败, 是否未在编码规则配置表中配置", "com.meicloud.cpf.config.aspect.codeGenError");
            }
            return code;
        } catch (Exception e) {
            throw new ValidateException("编码生成失败", "fsop.customer.receipt.codeGenError");
        }
    }
