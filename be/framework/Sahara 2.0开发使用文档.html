# Sahara 2.0开发使用文档

# 整体结构

  • sahara-configuration: 跨模块相关配置维护，仅供sahara框架内部使用;
  • sahara-core: 通用定义，如异常、分页结构，常用工具类
  • sahara-orm: 基于Mybatis-Plus增强处理，分页自动绑定参数、驼峰支持等；
  • sahara-web：web相关处理，目前提供跨域支持，以及js长整形精度丢失处理等
  • sahara-security: 安全框架
  • Sahara-test: 测试集成相关，目前提供Mybatis-Plus mapper层分片测试能力

整体上基于Springboot 条件化配置的方式进行开发，实现功能按需加载或快速覆写，提升启动性能；

# sahara-core

# 功能描述

  • 提供通用定义，如顶级异常、分页结构等；
  • 提供部分常用静态工具类以bean注册，并持有相关对象，以供非bean类直接使用；
  • 提供常用工具类实现；

# 异常定义

SaharaException继承RuntimeException，作为后续所有项目的顶级异常，所有异常的定义都需要继承该异常，以便后续做统一异常处理；

# PagedResult

分页数据查询返回给前端的数据结构

# ApplicationContextProvider

描述：应用上下文持有类，用于提供应用上下文，同时附加了一些简单的应用上下文操作 使用场景：非Bean类情况下需要获取应用上下文可直接使用它获取；

TIP

Bean内部获取应用上下文直接注入ApplicationContext即可

# ApplicationConfigurationProvider

描述：应用配置持有类，用于提供应用配置，同时附加了一些简单的应用配置操作 使用场景：非Bean类情况下需要获取应用配置可直接使用它获取；

TIP

Bean内部获取应用配置直接注入Environment对象即可

# 线程池

  • ThreadPoolManager: 线程池管理类，业务需要定义自己的线程池，内部自动根据线程池key命名线程名,格式为key-thread-%d
  • DefaultThreadPoolUncaughtExceptionHandler：线程池默认统一异常捕获处理器，默认只记录日志，需业务实现自身需求实现（除非显式catch抛出异常，否则直接是拿不到线程异常的）
sahara:
  core:
    thread:
      thread-pools:
        - name: COMMON_POOL   # 线程池key
          core-size: 32       # 核心线程数
          max-size: 32        # 最大线程数
          queue-size: 100     # 任务队列数
          keep-alive-time: 3   # 线程空闲存活时间，默认3s
          policy：            # 驱逐策略，

# SaharaBaseMapper

继承BaseMapper，扩展自有Mapper方法（后续根据需要继续增强），项目所有Mapper接口必须继承SaharaBaseMapper，不再继承BaseMapper；

现有扩展方法：

// 按据条件删除
int deleteByMap(@Param(Constants.WRAPPER) Map<String, Object> columnMap);
// 按条件查询
List<T> selectByMap(@Param(Constants.WRAPPER) Map<String, Object> columnMap);
// 更新所有字段，不管是否为空
int alwaysUpdateSomeColumnById(@Param(Constants.ENTITY) T entity);  
// 根据 entity 条件，查询一条记录（相比于selectOne，即使有多条记录，也不会报错）

  • Mybatis-Plus默认分页必须要使用@Param注解绑定参数，而且Mapper xml文件必须使用el表达式进行参数映射;
  • Mybatis-Plus底层逻辑写死必须分页必须使用IPage及其实现类或者List及其实现类进行分页（MybatisMapperMethod）；
  • Mybatis-Plus不支持RowBounds物理分页（NullPointException && 分页插件不支持）；
  • Mybatis-Plus分页相关代码侵入Mapper接口代理类，无法直接使用分页插件直接扩展；

示例如下：

      IPage<EmpEmployee> pageQuery(PageParam<EmpEmployee> page, EmpEmployee empEmployee);
}

# 使用BatchMapperHelper替代IService

IService本质只是Mapper批量操作方法，从职责分明的角度来看，不应该使用业务Service实现IService或者继承ServiceImpl，那么为什么Mybatis-Plus作者为什么这么做呢？ 其实也是无奈之举，批量操作方法涉及根据频次刷新内存，BaseMapper的那一套没办法取到频次入参，直接提供不刷新内存的批量操作方法又会出现性能甚至内存溢出问题（作者解释）.

目前Orm框架这边封装BatchMapperHelper工具类实现批量操作，同事也不直接支持在BaseMapper提供批量操作方法

# js长整形精度支持

JS能精准表示的最大整数是 Math.pow(2, 53)，十进制即 9007199254740992，也就是说长整形超过16位就会发生精度丢失情况，而我们的Long主键是20位，会发生精度丢失情况，因此将长整形序列化输出为字符串类型；

# 考虑到后续主键都是以长整形为主，默认开启
sahara:
  web:
    json:
      long-to-string: true

# sahara-security

# 登录密码加密/解密

  1. 前端请求sahara.server.public-key-url对应的url，拦截器拦截相应的url请求，返回公钥；
  2. 用户填完账号密码后，前端对密码进行加密，返回后端；
  3. 后端根据私钥进行解密后登录

# 单点登录参考配置

sahara:
  security:
    server:
      enabled: true
      logout:
        destroy-session: true
        central-logout: true  # 清除单点服务器session
    clients: CasClient   # 单点登录client
    cas:
      enabled: true
      login-url: https://siam.tqls.cn/login  # 单点登录页面地址
      prefix-url: https://siam.tqls.cn/         # 单点登录地址前缀，校验ticket/登出等需要用到
      callback-url: https://xxx.tqls.cn/backend/callback   # 单点服务器回调地址（返回ticket）
      protocol: CAS20   # 协议，一般情况不用管
    default-url: https://xxx.tqls.cn/#/ihr/home    # 登录成功后默认跳转地址 
    logout-default-url: 'https://siam.tqls.cn/login?service=https://xxx.tqls.cn/backend/callback?client_name=CasClient'   # 登出跳转地址

    @Resource
    private UserMapper userMapper;

    @Test
    public void deleteByName() {
        Assert.assertEquals(0, userMapper.deleteByName("NotExist"));
        Assert.assertEquals(1, userMapper.deleteByName("user1"));
    }
