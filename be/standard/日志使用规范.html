# 日志使用规范

# 日志级别的使用

  • ERROR

    ERROR级别的日志反映系统发生了非常严重的故障，无法自动恢复到正常态工作，需要人工介入处理。系统需要将错误相关痕迹以及错误细节记录ERROR日志中，方便后续人工回溯解决。

  • WARN

    WARN是低级别异常日志，反映系统在业务处理时触发了异常流程，但系统可恢复到正常态，下一次业务可以正常执行。当功能逻辑走入异常逻辑时，应该考虑记录WARN日志。

  • INFO

    INFO日志主要记录系统关键信息，旨在保留系统正常工作期间关键运行指标，开发人员可以将初始化系统配置、业务状态变化信息，或者用户业务流程中的核心处理记录到INFO日志中，方便日常运维工作以及错误回溯时上下文场景复现。

  • DEBUG

    DEBUG日志t通常用于开发人员记录各类详细信息，主要起到调试的作用，包括参数信息，调试细节信息，返回值信息等等。其他等级不方便显示的信息都可以通过DEBUG日志来记录。

# 添加日志的区域

  1. 代码抛出异常：凡代码抛出异常，不管后续是catch掉还是继续把异常抛出，都需要输出详细错误日志，以便后续追溯；
  2. 系统核心流程，业务关键动作
  3. 多分支判断：多if/else代码区域，建议记录分支判断参数和结果；
  4. 数据获取/处理不在预期范围
  5. 业务流程预期不符：
  6. 外部接口：对外部接口提供尽可能多的日志信息，有利于协助第三方快速排查定位故障；
  7. 系统初始化：核心模块或者组件初始化过程中往往依赖一些关键配置，建议在系统初始化时打印相关配置参数。

# 日志内容

一条完整的日志应该是一件事情的概要描述，其中应该的元素应该包括：

  • 日志级别, ERROR/WRAN/INFO/DEBUG
  • 日志时间
  • 概要内容，包括场景信息（什么功能等），状态信息(开始，中断，结束)以及重要参数.
  • 应用标识，方便日志搜索
  • 会话标识，能知道是哪个客户端或者是哪个用户触发，登陆账号，seesion信息等
  • 功能标识，方便日志搜索，跟踪指定功能的完整轨迹
  • 其他信息，其他可能的有用信息包括：代码位置，版本号，线程号等等

建议日志输出格式如下：

# 除了日志级别和概要内容，其他由系统自动完成
[日志级别][日志时间][应用标识][请求标识][session-id][线程id][代码位置][概要内容]

# 应该避免的错误

# 日志丢失

  • 禁止使用System.out.println("")打印日志，System.out输出的日志只会出现在控制台，不会被记录到日志文件;
  • 禁止打印异常日志不将异常对象exception返回，exception不返回无法获取异常调用堆栈信息；

# 性能浪费

  • 禁止使用字符串拼接，必须使用占位符方式，以节省某些时候不必要的拼接操作
  • 凡日志打印必有损耗，只打印有必要的日志

# 直接使用日志系统API

  • 禁止直接使用日志系统（Log4j、Logback）中的API，必须使用日志框架SLF4J中的API，使用门面模式的日志框架，有利于维护和各个类的日志处理方式统一；

推荐直接使用lombak的@Slf4j

# Java日志代码推荐示例

@Slf4j
public class XXX {
  public void test(String param) {
    log.debug("参数param值为[{}]"， param);  // 占位符方式，并且用[]包围，便于后续日志搜索
    try {
        // ...
    } catch(IOException e) {                  // 根据实际情况catch对应异常类型
      log.error("读取文件时发生错误。", e);   // 保留异常堆栈
    }
  }
}

# 日志运维

  • 所有日志至少保留15天，以便后续排查问题需要；
  • 日志需输出到ELK，方便日志快速搜索；
