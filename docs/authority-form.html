    # 表单权限

    依托于 i-form-item 我们不需要逐个在表单项中的表单控件上绑定 v-if/v-show、disabled 来实现隐藏和禁用，因为不仅 i-form-item 提供了 hidden/disabled 实现了同样的效果，而且 i-form-x 更是提供了 grants 字段用于全局接管 i-form-item 的这两个属性。进一步了解

    # 问题

    在真实的业务场景中依然存在如下问题：

    1. 后端返回的权限数据并不一定与组件需要的一致，那就意味着需要一个统一的入口作为转换层
    2. 一般来说权限配置并不会由开发者显式地在代码中申明，不仅具备太强的侵入性，导致本身就很复杂的业务表单跟表单权限逻辑造成冗余，而且难以维护需要逐个请求和传递权限信息

    # 方案

    我们如何做到兼顾如上的问题同时，让开发者对表单权限的逻辑无感知，让业务更纯粹呢？我们的实践方案是：表单权限信息由系统统一请求、转换、存储（全局状态：Vuex 或 Vue 原型链等）。每个 i-form-x 表单都显式地绑定 code 编码作为唯一标示。初始化时组件会根据 code 在全局状态中获取对应的权限配置，从而控制表单隐藏和禁用。

    解决了什么问题：

    1. 在系统请求后执行数据转换，适配了后端接口和组件属性不匹配的问题
    2. 只需要绑定 code 编码，以极低的“侵入性”实现业务和权限分离，不再需要开发者自行绑定对应的权限信息到 grants/hidden/disabled 字段

    # 实现

    最终的目标就是实现「如何让表单自动获取存储在全局状态中的权限配置」。首先组件本身并不具备这样的业务能力，我们可以通过 extends 或 mixins 的形式将通用表单拓展为适用于特定项目的通用表单。

    const FormX = {
  mixins: [Sahara.Form.X],
  // ...拓展
}

    我们知道 i-form-x 会调用 initGrants 将外界传入的 grants 字段转换传递至 grantsMap 用于后续的计算。源码如下：

    // 挂载流程权限配置
initGrants () {
  this.grantsMap = this.grants || {}
}

    initGrants 会分别在表单初始化时、grants 变化时、表单动态渲染时触发。这是很好的突破口，我们仅仅需要改写此处的 initGrants 函数即可，将 grantsMap 的来源改成全局状态对象，并根据自身的 code 获取指定的权限配置。

    import Vue from 'vue'
import Sahara from '@ipeople/sahara'

Vue.use(Sahara)

 // 请求、转换、存储权限信息至全局状态
const globalGrants = fetchGrants().then(formatGrants)

// 拓展 i-form-x
Vue.component(Sahara.Form.X.name, {
  mixins: [Sahara.Form.X],

  methods: {
    initGrants () {
      this.grantsMap = globalGrants[this.code] || {}
    }
  }
})
